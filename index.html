Getting set up

You don't need to be a developer — you just follow the steps and let Claude Code do the work. Here's the whole path, and where your team steps in.

1 · Put your project on GitHub

  1. Register on GitHub with your company email, then tell Imran or Oksana so they can add you to the company organization.
  2. Sign in with that same company email.
  3. Create an empty repository inside the company organization. Leave it completely empty — no README, .gitignore, license, or starter files. Those get prepared from your actual project.
  4. Copy the repository link — e.g. https://github.com/company-name/project-name.git
  5. Open the project folder in Claude Code. Open the main project folder itself, not a parent like Desktop, Downloads, or Documents.
  6. Paste the prompt below into Claude Code and add your repository link at the end.
Paste into Claude Code
I need to add this project to the company GitHub.

Please review the folder first and check for any sensitive files, passwords, API keys, temporary files, private client files, or anything else that should not be uploaded.

Then prepare the project with a README.md and .gitignore.

After that, help me initialize Git, create the first commit, connect it to the correct company GitHub repository, and push it to the main branch.

Please explain each step in simple language before doing it.

GitHub repository link:
[paste the repository link here]

Never upload

  • Passwords, API keys, .env files
  • Private client files, personal files, temporary files
  • Any confidential company files that aren't part of the project

2 · What the agent builds — the house defaults

When you ask for backend work, Claude Code sticks to one predictable, lightweight stack so every project looks familiar and nothing gets over-engineered.

Default stack

  • Python + FastAPI (when an API is needed)
  • SQLAlchemy for the database
  • SQLite for lightweight data, cloud MySQL for production
  • Docker Compose — backend container only
  • Ruff (formatting/linting), Pytest (tests)
  • Alembic only when the schema will change

Deliberately avoided

  • Celery, Redis, Kafka, Kubernetes
  • A local MySQL container (MySQL is cloud-hosted)
  • Dependency-injection frameworks
  • Microservice splits for simple services
  • Deprecated or unmaintained packages
  • Extra infrastructure without a clear reason

3 · What your team handles (don't do these alone)

A few things are coordination points, not something you or the agent should guess at. When you hit one, the agent will pause and tell you to reach out.

Ask DX for

Cloud MySQL access — host, port, database name, username, password or secret reference, SSL requirements, and any VPN / allowed-IP rules. The agent will never invent database credentials; it asks for them.

Ask Imran or Oksana for

Access to the company GitHub organization before you create your repository.

Still being finalized

Environment-specific setup (development, staging, production) and where production secrets live. Treat these as team decisions — flag them and wait rather than hardcoding anything.

4 · The one rule that never bends

Secrets stay out of the repo

Real passwords, keys, and tokens live only in a local .env file that never gets uploaded. The project commits a safe .env.example with placeholder values instead. The kit's settings even block the agent from reading your real .env files.
